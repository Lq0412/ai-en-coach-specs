# PR Review 规范

PR 的**创建格式**见 [pr-commit](../pr-commit/SKILL.md)（四段描述模板）。本文只管 Review 和合入这半段。

## 核心原则

**不经过 Review 的代码不能合入。** AI 生成的代码同样走完整 Review，Review 时需确认代码逻辑清晰可理解。

## Review 流程

```
PR 提交 → 指定 Reviewer → 24h 内给反馈
              ↓
     ┌── 通过 → 合入
     │
     └── 需要改 → author 改完 → 再 Review → 通过 → 合入
```

### 设计草案 PR（不合并）

产品设计文档、原型等设计产出可通过 PR 进行逐行评审。**此类 PR 只用于 Review、不合并**：讨论定稿后把内容落回对应 Issue，并在 Issue 中附上该 PR 链接。此后 Issue 作为开发起点与后续跟踪。

（系统架构的空骨架 PR 会合并入库，与设计草案 PR 不同。）

### 时效

| 节点 | AI 行为 |
|:-----|:-----|
| Review 反馈 | PR 分配后立即审查，输出检查结果 |
| 作者改完 | 检测到新提交后自动重新审查差异 |
| 再 Review | 改动后即时确认 |

### Review 分工

| 角色 | 职责 |
|:-----|:-----|
| **项目 Lead** | 关键 PR 或关键技术风险上提供反馈 |
| **指定 Reviewer** | 日常 PR 的方向与改动可理解性（是否对应 Issue、改动能否讲清） |
| **团队成员** | 鼓励互相 Review |

## Review 检查项

Review 不是「我看了一遍代码」。按以下维度逐条过：

### 1. 正确性（必审）

- 能不能正常工作？边界情况处理了吗？
- 有没有遗漏的错误处理？
- 会不会对已有功能产生副作用？

### 2. 安全性（必审）

- 有没有硬编码的密钥、token？
- 用户输入校验了吗？
- 授权检查到位了吗？

### 3. 架构（必审）

- 改动的文件放在正确的模块/目录下了吗？
- 有没有破坏既有的模块边界？（A 模块直接访问了 B 模块的内部实现）
- 数据模型改动是否兼容既有的数据？

### 4. 可维护性

- 新人对这段代码能在 5 分钟内看懂吗？
- 注释写清楚了「为什么这样做」而不是「做了什么」吗？

### 5. AI 代码专项检查

如果 PR 包含 AI 生成的代码，追加三条：

- [ ] 询问提交者：是否能讲清每一段 AI 生成代码的逻辑（不只是「大概是这个意思」）
- [ ] 检查是否有 AI 常见的糊弄行为（兜底数据、猜测字段、冗余异常处理）
- [ ] 检查核心 PR 是否附了 Prompt 和人工 Review 说明

## 合入条件

满足**全部 5 条**才能合：

1. 至少 1 人 **Approved**
2. 自动化验证通过（编译无错误、测试通过、lint 无新增告警）
3. 已 rebase 到最新目标分支，无冲突
4. 所有 Review 意见已处理（改了或说明了为什么不改）
5. PR 改动范围与关联 Issue 一致，无夹带无关修改

## Review 意见怎么写

### 好意见

```
L42 这里如果 audioStream 为 null，会直接崩掉。
改成 try/catch 或者在调用方加 null 检查？
```

特点：**指出位置 + 说清问题 + 给方向不代写**

### 差意见

```
感觉这里不太对
这段代码写得太复杂了
```

问题：**没说具体位置、没说为什么不对、没给方向**

### 按严重度区分

| 前缀 | 含义 | 不修会？ |
|:-----|:-----|:--------|
| `[阻塞]` | 不修不能合 | 功能崩、数据丢、安全漏洞 |
| `[建议]` | 不修可以合，但改了更好 | 可读性、性能微优化 |
| `[疑问]` | 没看懂，请教一下 | — |

## 合入后

- PR 合入即执行，不堆积
- 功能分支立刻删
- 如果合入后发现问题 → 新开 fix 分支修，不回滚已合入的代码（除非是阻塞级 bug）
